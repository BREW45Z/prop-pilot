# Product Principles

## Purpose

These principles exist to keep Prop Pilot focused, consistent, and trustworthy as it grows. They should guide every product, design, and engineering decision.

---

## Principle 1 — Solve Real Trading Problems

Every feature must solve an actual problem experienced by prop traders.
Do not add features simply because competitors have them.

---

## Principle 2 — Accuracy Before Features

Calculation accuracy is always more important than shipping new functionality.
If a calculation cannot be trusted, it should not be released.

---

## Principle 3 — Simplicity Wins

Prefer the simplest solution that delivers value.
Avoid unnecessary complexity in both the interface and the implementation.

---

## Principle 4 — One Clear Purpose Per Screen

Each page should have one primary objective.
Users should immediately understand what they can accomplish.

---

## Principle 5 — Speed Matters

The application should feel instant.
Minimize clicks, loading, and unnecessary interactions.

---

## Principle 6 — Professional Design

Maintain a clean, modern, and distraction-free interface.
Use whitespace, typography, and hierarchy to improve clarity.

---

## Principle 7 — Mobile First

Every feature must work well on mobile devices before considering larger screens.

---

## Principle 8 — Consistency

Use the same spacing, colors, terminology, icons, and interaction patterns throughout the product.

---

## Principle 9 — Educate While Assisting

Whenever appropriate, explain calculations, rules, and trading concepts without overwhelming the user.

---

## Principle 10 — Build for Daily Use

Every feature should encourage traders to return before, during, or after each trading session.

---

## Product Decision Checklist

Before building any new feature, answer these questions:

- Does it solve a real trading problem?
- Is it accurate?
- Is it easy to understand?
- Is it fast to use?
- Does it fit the existing design language?
- Would traders use it every day?
- Does it make Prop Pilot more valuable without making it more complicated?

If any answer is "No", reconsider or redesign the feature.

---

## Definition of Quality

A feature is considered complete only when:

- The calculations are verified.
- Edge cases are handled.
- The UI is responsive.
- The design matches the design system.
- Documentation is updated.
- Testing has been completed.
- The feature genuinely improves the trader's workflow.

---

## Closing Statement

Prop Pilot is not built to have the most features. It is built to provide the most confidence. Every decision should reinforce trust, simplicity, and precision.
